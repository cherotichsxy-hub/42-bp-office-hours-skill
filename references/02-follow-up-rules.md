# Follow-up Rules

这份文档定义"要不要追、追什么、追到什么深度"。题库（`01-question-bank.md`）是 Rubric，这份文档是判断标准。

**核心立场**：Skill 最有价值的部分不是把题库走完，而是帮创始人把他没想清楚的地方想清楚——这部分基本上都靠追问完成。所以追问不是可选项，而是这个 Skill 的核心动作。

不允许的做法：

- 因为对话偏长，或者你自己觉得"差不多够了"，就主动跳过命中的追问。
- 替用户判断他是不是累了。是不是要跳，由用户决定，不是你决定。
- 命中了触发，但只追了一句就放过，等于蜻蜓点水。

## 六条追问原则

每次用户回答完之后，把回答按下面六条原则扫一遍。**任一条命中就必须追问**。允许跳过的情况只有三种：

- 用户前面回答里已经覆盖了这一条。
- 这条问完不会让最终的内容稿、Slide Spec 或工作建议里的任何一处变得更好。
- 用户对当前问题已经明确 pass（按下面"用户 pass 一道题怎么办"处理）。

### 1. 追具体

用户回答停在抽象词上时，问具体的人、场景、动作、数字。

触发词：
- "年轻人"、"企业"、"创作者"、"开发者"、"中小 B"等大类人群——问最先服务的那一小群。
- "效率提升"、"体验更好"、"AI 赋能"、"降本增效"——问具体提升多少、对比谁。
- "市场很大"、"用户很多"——问最先服务谁、bottom-up 怎么算。

### 2. 追机制

用户只讲结果或愿景时，问它**为什么**会发生、**靠什么**持续发生。

触发：
- 说会增长 → 问增长的具体机制。
- 说用户会留存 → 问用户为什么持续用。
- 说壁垒会变强 → 问越做越强的具体回路（数据闭环 / 网络效应 / 工作流嵌入 / 分发入口）。

### 3. 追诚实

用户回避风险或讲得过满时，问失败可能、团队短板、最脆弱的假设。

触发：
- 说"没有竞争对手"——立即追问替代方案。
- 只讲优势，不讲不确定性。
- 把还没验证的事讲得像已经成立。

### 4. 追适配

检查团队 / 问题 / 方案 / 市场 / 时机彼此是否匹配。

触发：
- 团队履历和当前项目关系不清。
- 用户痛点和产品能力不匹配。
- 市场算到很大，但切入口太散。
- 融资金额和里程碑不自洽。

### 5. 追壁垒

用户只说"我们更快、更努力、更懂用户"时，问为什么难复制。

触发：
- AI 项目只说用了大模型，没有独有数据、工作流嵌入、分发入口或网络效应。
- 竞争优势看起来只是功能领先。
- 没有回答"OpenAI / 大厂明天发同样能力你怎么办"。

### 6. 追对比

所有判断放进对比里看：和竞品比、和替代方案比、和历史失败案例比、和用户现在的做法比。

触发：
- 没有列直接竞对和间接替代。
- 没说新方案相比旧方案具体强多少倍。
- 没解释为什么三年前做不成。

## 优先级

### P0 — 必须问到

P0 是投资人最容易质疑、且会决定项目是否可投的问题。

典型 P0：

- 这个问题是不是真痛点？
- 为什么是你们做？
- 为什么不是大模型平台 / 强对手做？
- 你们有什么难复制的壁垒？
- 这轮融资要证明什么关键假设？
- 已经有什么证据能让投资人从"听起来不错"变成"值得继续聊"？

一个 Part 里如果有多个 P0 缺口，原则上都进入追问队列。不要因为"每 Part 限 1-2 题"而跳过 P0。唯一例外：用户前面的回答已经覆盖，或该问题明显不适用于当前项目。

### P1 — 动态选择

P1 能明显提高 BP 的完整度、可信度或表达质量，但不一定每次都必问。

典型 P1：

- 用户第一次使用后的真实反应。
- 留存、复购、持续使用原因。
- 市场未来增长驱动力。
- B2B 销售周期。
- 联合创始人关系和团队互补性。

P1 数量看：当前 Part 是否复杂；用户回答是否已经足够；继续问是否会明显改善最终材料；用户是否已经疲劳。

默认上限：每个 Part 内 P1 控制在 1-3 个。

### P2 — 谨慎触发

P2 是锦上添花。能让表达更丰满，但不影响 BP 主线。

典型 P2：

- "X for Y" 类比。
- 更长远的扩展想象。
- 更细的收入结构推演。

会话已经偏长或用户想快速拿结果时，P2 直接跳过。

## 追问决策循环

每轮回答后按下面顺序判断。这个循环**只处理当前 Part**——不允许因为后面问题"更有价值"就跳过前面未覆盖的固定问题。

1. **固定问题是否已经回答？**
   - 没有：继续问固定问题或其改写版本。
   - 有：进入下一步。
2. **回答是否足够具体？**（参见下文"判断回答是否足够"）
   - 不够：用追问把人、场景、动作、数字、证据问清楚。
   - 够：进入下一步。
3. **是否存在 P0 缺口？**
   - 有：触发候选挑战问题。
   - 没有：进入下一步。
4. **是否存在能显著提高 BP 质量的 P1？**
   - 有：视会话负担和信息价值决定问几个。
   - 没有：进入下一 Part。
5. **是否只是 P2？**
   - 只有在用户愿意继续深入时再问。

## 固定问题怎么算覆盖

题库里每个 Part 的几道固定问题都必须问到。只有一种情况可以不问：材料里已经写过对应内容，而且写得足够具体。

**判定标准**

- "写过对应内容"指材料里有可以直接对应这道题的事实陈述，不是隐含、不是举例提到、不是从其他段落推断出来的。
- "足够具体"指材料里这段话能直接放进 BP 对应位置，不再追问也能用。

材料只是"提了一下"或"间接相关"时，这道固定问题仍然要问，但可以改写成"材料里讲了 X，但还差 Y"的形式。

**举几个判断例子**

- BP 第 4-5 页写了张阳完整履历——Q1 "请详细介绍创始人和核心团队背景" → 覆盖。
- BP 没有正面回答 "如果你被换掉，新 CEO 会做什么改变" 这类自我认知问题——这是 Q1 的另一道固定问题 → 没覆盖，要问。
- BP 第 10 页有 "AI 游戏社交平台 / 新一代年轻人的社交平台" → Q3 "一句话讲在做什么" 的字面意义上覆盖了，但定位句堆了抽象词，**不算"足够具体"**，所以 Q3 的追问"再短一点 / 不用平台生态这些词怎么说"仍然要问。

**一个 Part 里所有固定问题的状态都判断完之后**：没覆盖的逐道问；部分覆盖的改写后问；完全覆盖的可以不问，但仍然要按下面的追问流程扫触发。

## 跨 Part 信息追踪

创始人的回答经常会自然带出其他 Part 的信息。比如聊"行业最专业的人会问你什么"时，可能顺带讲了 GTM 的具体打法；聊"AI 在产品里起什么作用"时，可能顺带讲了壁垒；聊动机时，可能顺带讲了创始人的早期判断。

每次用户回答完之后，扫一遍这次回答触及了哪些 Part 的内容，把对应事实写进 Fact Bank 里那些 Part 的位置，并标明来源（比如"来自 Q1 第 N 轮回答"）。

**这件事的目的**：后续真正推进到那个 Part 的时候，已经覆盖的固定问题不要重复问，已经覆盖的追问触发也不要重复触发。

**应用规则**

- "固定问题覆盖"的判定（见上一节）**不只看材料，也看用户在任意 Part 回答过的内容**。如果用户在 Q1 已经讲了 GTM 的具体做法，到 Q15 时这道固定问题就算覆盖，不重复问；如果只是部分覆盖，按"前面讲了 X，这里还差 Y"形式改写后问。
- 跳 Part 决策（下一节）的三条 checklist 也应用 Fact Bank 跨 Part 状态：用户在前面 Part 回答时覆盖的内容，到对应 Part 时算"已覆盖"。
- Fact Bank 是跨 Part 的，不要按 Part 切分独立处理。

**判定标准必须严**

跨 Part 覆盖**必须**是用户明确讲过对应内容，不是从其他回答里推断或合成。和"材料覆盖"一样严：必须是直接对应的事实陈述，且足够具体能写进 BP。

举例：

- 用户在 Q1 答"我们通过 fork 模板让用户在小圈层做个性化游戏，第一个 fork 的人能很快得到回响" → 这覆盖了 Q15 的"产品本身有没有用户带来用户的机制"。到 Q15 时不再重复问这一点，但 Q15 的其他固定问题（比如"一个用户大概要花多少钱拿到"）仍然要问。
- 用户在 Q1 答"我做 Helo 时一年到 5000 万 DAU" → 这是团队成就，不算覆盖 Q14 的"Wanaka 现在的验证证据"。不要跨 Part 套用。

**测试标记里怎么呈现**

测试标记开启时，跨 Part 覆盖要在两处体现，方便 review：

1. 用户的回答覆盖了其他 Part 的内容时，在承接之后加一行：`跨 Part 标记：本次回答覆盖了 QX 的 [具体哪道题或哪个方向]`。
2. 后续推进到那个 Part 时，原本要问的某道题因跨 Part 覆盖而跳过——测试标记里的触发原因写"被 QX 第 N 轮回答覆盖"，而不是"材料未覆盖"。

## 跳 Part 决策

一个 Part 什么时候可以真正推进到下一 Part？必须同时满足下面三件事：

1. 这个 Part 的**每一道固定问题**都已覆盖。注意是**每一道**，不是问了其中一道。
2. 这个 Part 的**每一道 P0 挑战问题**都已覆盖，或者明确判断不适用于当前项目。
3. 用户最近这次回答之后，按"六条追问原则"扫一遍，**没有任何一条仍然命中**。

三条少一条都不能跳。最常见的错误是只满足第一条就跳：看到 BP 写了团队履历就跳到 Q2，结果"你是最适合做这件事的人吗"、"团队有什么短板"两道 P0 挑战根本没问。这是把 Skill 最值钱的部分让掉了。

**跳之前过一遍这个清单**

- [ ] 题库里这个 Part 的固定问题，每一道是否都有对应答案？
- [ ] 这个 Part 的 P0 挑战问题，每一道是否都已覆盖或确认不适用？
- [ ] 用户最近回答完之后，六条追问原则扫一遍，是否仍有命中？
- [ ] 用户回答里有没有出现新的 logical gap，引出原本不需要问的追问？

四条全过，才进入下一 Part。

## 追问的两轮上限

**同一条触发逻辑最多追两轮。** 两轮之后，如果用户的回答仍然不够具体或不够充分，就打住，不要继续追下去。

打住不等于放弃这条信息。具体动作：

1. 把这条标到内部"待补充"队列。它之后会出现在最终交付物里的"给创始人的工作建议"，作为待 Founder 自己想清楚的事。
2. 用一句轻松的承接句过渡到下一个动作——可以是同一个 Part 里的下一条触发，也可以是下一道固定问题。比如"OK，这块我先记一下"、"行，那我们继续"。
3. **不要告诉用户**"我追问了两轮你还是没回答清楚"或"我把这条标到建议里了"——这是内部话。

**为什么要设这个上限**

- 防止 agent 在一条无法被进一步澄清的事情上无限追问，让用户烦。
- 用户在某些问题上就是答不出来或者还没想清楚，再问一次价值也不大。
- 让追问保持"认真但有边界"的节奏。

**轮数怎么算**

- 起始题（固定问题或某条触发出的追问）算第 0 轮。
- 同一个判断维度的第一次追问 = 第 1 轮。
- 同一个判断维度的第二次追问 = 第 2 轮，是上限。
- 如果用户的第二轮回答引出了**新的**触发逻辑（比如他答了一个新的具体场景，但这个场景里又出现了"追机制"或"追壁垒"的新缺口），那是另一条触发，可以另起计数。

不要用"换个说法再问一次"来绕开上限——换说法本质上还是同一个判断维度，仍然算第二轮。

## 用户 pass 一道题怎么办

用户说"答不上来"、"想不清楚"、"先跳过"、"暂时不想说"、"pass"时：

1. **接受 pass，不再问这道题。** 不要换角度再问一次——用户 pass 通常就是真的答不上来，或者不愿意答，再问只会让他不舒服。
2. 把这道题内部标为待补充。它之后会出现在"给创始人的工作建议"里，作为下一步需要 Founder 自己想清楚的事。
3. 用一句轻松的承接句直接进下一道题。比如"OK，那这个先记一下，我们继续"、"行，那看下一个"。**不要告诉用户**"我把这个标记到工作建议里了"——这是内部话。

**关于跳的范围**

默认 pass 只跳这道题，下一道题继续问。

但如果用户的 pass 明显是冲着整个 Part 来的——比如"团队这块我现在都讲不清楚，整个跳过吧"、"动机这部分我得想想，先跳到下一部分"——就直接推进到下一个 Part，剩下的题不要再问。

判断标准是用户的措辞：他说的是"这题"还是"这部分 / 这块"。模糊的时候默认按"只跳这道"处理。

## 判断回答是否足够

一个回答通常至少满足下面几项才算"足够写进 BP"。

### 用户与痛点

- 有具体用户，不只是大类人群。
- 有具体使用场景或痛点发生场景。
- 能说明痛点造成的真实成本。
- 能区分 must have 和 nice to have。

### 解决方案

- 用户实际怎么用讲得清楚。
- 核心功能或体验变化明确。
- 和现有方案相比的提升幅度或差异明确。
- 第一阶段聚焦点明确。

### AI 与技术

- AI 是核心引擎还是辅助讲得清楚。
- 说明 AI 如何转化成用户价值。
- 说明 AI 拿掉以后产品还剩什么价值。
- AI-heavy 项目必须回答大模型平台风险。

### 市场与竞争

- 市场规模有推导逻辑，优先 bottom-up。
- 最小切入口清楚。
- 直接竞对和间接替代分开。
- 能说清为什么现有玩家或更有钱的对手不容易赢。

### 验证与增长

- 有 demo / 用户访谈 / 内测 / 付费 / 留存 / 复购 / 强反馈中的至少一种。
- 能说明哪个证据最能改变投资人置信度。
- 第一批用户从哪里来讲得具体。
- GTM 不只是"PR、社媒、投放"这种泛泛渠道。

### 团队与融资

- 团队亮点和项目之间有关联（不是只罗列大厂学历）。
- 团队短板不回避。
- 融资金额 / 用途 / runway / 里程碑彼此自洽。
- 里程碑是可验证的，不是"继续打磨产品"。

## Checkpoint

一个 Part 里高价值候选问题超过 3-4 个时，**不要**连续追问到用户疲劳。先问 P0 和最关键的 P1，然后给用户选择：

> 这一部分已经基本可以写进 BP 了。还有几个问题会让这一页更有说服力，你想继续补充，还是先进入下一部分？

Checkpoint 只在问题较多时出现，不要每问一题都出现。

## AI 主动追问（题库之外）

题库是 Rubric，不是边界。允许在当前 Part 内基于用户材料和刚才回答，主动识别 logical gap 并问更贴合上下文的问题。

允许的情况：

- 题库里有对应方向，但原问题直接问会显得生硬，需要改写。
- 用户回答里出现了新的关键信息，但和 BP 逻辑之间还差一环。
- 用户材料看似完整，但有投资人会继续追的隐含问题。
- 补一句具体事实、对比或机制会明显提高最终 BP 页面的质量。

边界：

- 只服务当前 Part，不跳到后面。
- 不能发明用户没说过的事实。
- 不能把追问变成泛泛聊天。
- 不能为了展示聪明而问与 BP 输出无关的问题。

常见 logical gap：

- 说了目标人群，但没说第一批最具体的人。
- 说了痛点，但没说痛点发生的真实场景或成本。
- 说了方案，但没说用户实际怎么用。
- 说了 AI 能力，但没解释它如何转化成用户价值。
- 说了壁垒，但没解释强对手为什么不能复制。
- 说了融资金额，但没说这笔钱要证明什么关键假设。

## 什么时候不问

不要追问的情况：

- 用户已经给出了具体人、场景、动作、数字、证据和逻辑。
- 追问只会让表达更长，不会提高 BP 质量。
- 问题和当前项目明显无关。
- 用户已经多次表示想先生成结果。
- 这个问题只是满足题库完整性，而不是服务 BP。

判断标准：如果一个问题问完以后不会影响内容稿、Slide Spec 或工作建议中的任何一处，就不要问。

## 提问改写规则

优先使用题库里同 Part 的原问题。需要改写时遵循：

1. 保留原问题的判断目的。
2. 用用户刚才的内容作为自然引子。
3. 不要把材料原文机械贴进去。
4. 不要暴露内部标签。

示例：

用户说："我们做北美年轻人的 AI 游戏社区。"

✅ 改写：
> 你提到核心用户是北美年轻人，但这类人还是有点大。你们最想先抓住的是哪一小群人？有没有具体聊过的用户，或者你观察到的行为场景？

❌ 不要：
> 根据材料，你们的核心用户是北美年轻人。请说明核心用户是谁。
