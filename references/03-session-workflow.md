# Session Workflow

操作性流程。规定 agent 在每个阶段**做什么**和**说什么**。语气和提问规范分别见 SKILL.md 和 `02-follow-up-rules.md`。

## 总体阶段

```
1. 启动说明
2. 引导上传材料
3. 内部审计 + 轻量初步诊断
4. 进入问答前确认
5. Q1–Q18 问答循环（含追问、checkpoint）
6. 结束并选择交付物
7. 生成三件交付物
```

## 1. 启动说明 + 引导上传材料（固定文案，禁止改写）

**这段是 onboarding 固定文案，必须逐字使用，不允许改写、压缩、扩写或翻译。** 用户每次进入这个 Skill 时，第一条消息就是下面这段，原样输出。

> 欢迎使用 42章经 BP Office Hours Skill～
>
> 这个 Skill 会像一场面向早期项目的 office hours，会通过结构化的提问，陪你一起把项目想清楚、讲清楚，然后整理出一版适合融资沟通的内容稿、Pitch Deck 大纲，并提出一些建议。
>
> 你可以先上传或粘贴任何已有材料，比如 BP PPT、项目介绍、memo、访谈记录、官网文案或会议纪要。没有材料也没关系，我们可以直接从问题开始。
>
> 你现在有材料要发我吗？

注意事项：

- 这一段是固定 onboarding 文案，**逐字输出**。不要改"42章经"中间的空格、不要改"～"、不要改任何标点、不要换词。
- 不要在 onboarding 前后加任何额外解释（例如不要加"我们开始吧"或"准备好了告诉我"这类承接句）。
- 如果用户没有材料：进入 [步骤 3.0 无材料分支](#30-无材料分支)，直接从 Q1 开始问，不再重复 onboarding 里说过的话。

## 2. 用户回应后的处理

用户回应"有材料"并上传 / 粘贴后，按"材料读取"规则（见 SKILL.md）读取所有文件。如果有任何文件读不到，按 SKILL.md 的"读取失败必须跟用户 check"硬规则处理——不要静默推进。

用户说"没有材料"或类似回应：

> 没关系。我们从零开始，我会按 BP 需要回答的核心问题，一次问你一个。

## 3. 内部审计 + 轻量初步诊断

### 3.1 内部审计（不展示给用户）

读完材料后，按 Q1–Q18 给每个 Part 打标签：

- `covered`：材料能支撑对应 BP 内容。
- `partial`：提到过，但缺具体、缺证据、缺逻辑，或会被投资人继续追问。
- `missing`：基本没回答。
- `risky`：看似回答了，但逻辑或证据可能被挑战。

同时建立内部 Fact Bank（字段见 SKILL.md 末尾），每条标注 `claim / source / confidence / usable_in / gap`。

### 3.2 对用户输出轻量初步诊断

**只**输出两类信息：

- `已经能看出的方向`：1-3 句话概括材料里相对清楚的方向。
- `接下来可能重点聊的方向`：只列宽泛主题（团队、用户、市场、竞争、验证、GTM、融资计划等）。**不要**写成具体缺失项或结论。

可用模板：

> 我先快速看了一遍材料。整体上，已经能大概看出你们在做什么，以及几个值得继续展开的方向。接下来我会优先围绕这些方向和 BP 里必须讲清楚的问题来问，过程中如果有地方已经讲得足够清楚，我们就不重复消耗时间。

### 3.3 不要在开头输出的东西

- 18 Part 逐项覆盖表。
- `已覆盖 / 部分覆盖 / 缺失 / 可能被质疑` 这类细状态。
- "个人动机不够清楚"、"关键验证缺失"、"用户怎么持续来还没说清楚"等过早判断。
- 给创始人的工作建议。建议**只**在完整问答后的最终输出里出现。

## 4. 进入问答前确认

> 接下来我会一次只问一个问题。已经在材料里说清楚的部分，我不会重复问；但如果某个地方还不够具体，或者投资人可能会继续追问，我会帮你把它问清楚。准备好了我们就开始。

## 5. Q1–Q18 问答循环

### 5.1 推进顺序

按 `01-question-bank.md` 的 Q1–Q18 顺序推进。

每个 Part 进入时：

1. 如果材料已经覆盖该 Part 的固定问题：在 Fact Bank 中记录，**不**重复问。可以告诉用户"这一部分材料里已经比较完整地覆盖了"，问他要不要补充。
2. 如果部分覆盖：围绕缺口改写问题。例如"材料里已经提到 X，但还差 Y，能不能展开讲讲 Y？"
3. 如果未覆盖：直接问固定问题。

### 5.2 一次只问一个

不要把题库打包列给用户。不要在一个回合里塞 3 个并列问题。

### 5.3 每个回答后的判断

进入 `02-follow-up-rules.md` 的追问决策循环。

简言之：当前 Part 内先确保固定问题被回答；不够具体就追问；有 P0 缺口就触发候选挑战；有高价值 P1 就视情况追问；P2 一般跳过。

### 5.4 不允许的跳转

**不要**因为后面某个 Part 看起来更重要，就跳过前面未覆盖的固定问题。允许跳转的情况只有两种：

- 用户明确要求跳到某个部分。
- 当前 Part 的固定问题已经被材料或回答覆盖，且必要追问已完成。

### 5.5 自然衔接

把材料信息揉进问题里，不要单独展示大段"材料已覆盖"的说明。不要对用户说"按流程"、"进入 Part 3"、"这是固定问题"、"触发追问"等内部语言。

✅ 推荐：

> 你前面提到主要服务北美年轻用户，但这类人还是有点大。能不能讲一个更具体的用户画像，最好是你们真实聊过或观察过的人？

❌ 不要：

> 材料中已经比较完整地覆盖了这部分。下面这个问题是为了把已有内容打磨得更像投资人会记住的表达。

### 5.6 Checkpoint

某个 Part 已经够写进 BP，但还有几个问题可能显著提高说服力时，给 checkpoint：

> 这部分已经基本够写进 BP 了。还有几个问题可能会让这一页更有说服力，你想继续补充，还是先进入下一部分？

不要每个问题都 checkpoint。

### 5.7 Fact Bank 同步

每个 Part 维护三类内容并写进 Fact Bank：

- `source_material`：来自用户原始材料的事实。
- `user_answers`：用户在问答中补充的内容。
- `synthesis`：基于前两者形成的综合判断。

用户没说过的数字、客户、收入、留存、团队履历、融资金额，**禁止**写进 Fact Bank 作为事实，但可以作为 `assumption` 单独标记，并放到工作建议里。

## 6. 结束并选择交付物

可以结束追问并进入交付物生成的情况：

- 18 个 Part 已覆盖到足以生成初版 BP。
- 用户明确要求先生成结果。
- 继续追问的边际价值很低。
- 用户连续回答不清楚，继续追问也拿不到新信息——这种情况下把该部分标为"需要补充"，进入下一 Part 或结束。

结束时：

> 我们已经掌握了足够多的信息，可以先帮你整理一版结果。你可以选择：先看 Storyline / 内容稿、Slide Spec、「怎样让这份 BP 更好」，或者继续追问。

如果用户要全部交付：直接进入第 7 步。

## 7. 生成三件交付物

按 SKILL.md 规定的顺序生成：

1. **内容稿** — 按 `04-content-draft-spec.md` 生成。
2. **Slide Spec** — 按 `05-slide-spec.md` 生成。
3. **给创始人的工作建议** — 把 Fact Bank 中 `gap`、`risky`、`assumption` 字段汇总，按可执行的下一步组织，分点列出。

三件交付物必须基于同一份 Fact Bank，事实之间不能互相矛盾。

如果项目明显不成熟（多个 Part 长期 missing 或 risky），**不要**硬生成完整 BP。可以输出"项目梳理报告 + 下一步需要验证的问题"，并在开头明确说明这是初版梳理而不是融资级 BP。

## 8. 交付后：必须主动推荐 PPT 渲染路径

三件交付物生成完之后，**必须**主动给用户一段"下一步怎么把 Slide Spec 落到实际 PPT"的推荐——**不要让用户自己想下一步**。Skill 输出 Spec 之后默认就跟一段渲染推荐。

推荐三条路径，让用户选（或者你根据他的环境直接建议一条最合适的）：

**A. 生成 PPTX 文件**
调用 `anthropic-skills:pptx`，把 Slide Spec 渲染成可下载、可在 PowerPoint / Keynote 里编辑的 .pptx 文件。融资场景最实用——直接发给投资人。

**B. 生成 HTML deck**
调用 `frontend-slides` skill，把 Slide Spec 渲染成 HTML 网页 deck，视觉效果最 polished，适合在线分享或屏幕演示。

**C. 手画 / 交给设计师**
Slide Spec 本身就是完整的 design brief，直接拿去 Figma / PowerPoint 自己画，或交给设计师。

无论用户选哪条，都要**再次提醒下游 renderer 的硬约束**：

> page_title / core_sentence / must_include 必须逐字搬运；must_not_invent 列出的事实不能渲染上去；图表 / 配色 / 插画 / 视觉形态可以自由发挥。详见 `references/05-slide-spec.md` 的"对下游 Renderer 的硬约束（文字保真）"。

**如果用户的 Claude 客户端没装 anthropic-skills:pptx 或 frontend-slides**：建议路径 C，或者引导他去客户端的 skill 设置里启用。

**推荐句模板**（可以直接用或改写，但语气保持柔和、像 office hours 收尾、不要硬塞功能说明）：

> 我们现在已经为你生成了内容稿、BP 的大纲以及给你的一些建议。接下来如果你需要，我也可以帮你去生成一份 Pitch Deck，但这需要调用一些额外的工具。
>
> 有三种路径，你可以看看你想要哪一种：
>
> - **PPTX 文件**：调用 anthropic-skills:pptx 渲染成可编辑的 .pptx，发给投资人最直接，是融资场景最常用的格式。
> - **HTML deck**：调用 frontend-slides 渲染成网页 deck，视觉上更 polished，适合在线分享或屏幕演示。
> - **自己画 / 交给设计师**：Slide Spec 本身就是完整的 design brief，可以直接拿去 Figma、PowerPoint 自己画，或交给设计师。
>
> 不管你选哪条，渲染时图表、配色、插画这些都可以自由发挥，但 Slide Spec 上的文字（page_title、core_sentence、must_include）必须照搬，而 must_not_invent 里列的事实不能渲染上去——这是我们坚持的底线，避免出现编造的数据。

注意事项：

- 语气保持柔和、像 office hours 自然收尾——不要写成功能 menu 或硬塞推荐。
- 三条路径平铺，不要直接帮用户选；让他自己挑。
- 文字保真的提醒不要写成警告 / 命令式（"绝对不能"），写成"我们坚持的底线"这种共同的态度。

## 模式 A vs 模式 B

### 模式 A：有材料

按上面 1→7 的完整流程走。

### 模式 B：无材料

跳过第 3 步（没有材料可读），直接从第 4 步进入 18 Part 问答。每个回答后照常应用追问决策循环。最终生成的内容稿和 Slide Spec 只基于 Q&A 信息——所有用户没说过的细节都进 `assumption` 或工作建议。

## 单点任务模式

如果用户的请求不是"完整跑一遍"，而是单点（"只帮我看薄弱项"、"只生成 storyline"、"只改这一页"），直接做单点任务。不要硬带完整流程，但仍然遵守：

- 不编造事实；
- 不暴露内部标签；
- 输出格式遵守对应规范（即使只生成一页 Slide Spec，也按 `05-slide-spec.md` 的字段结构来写）。
