# 内容稿规范

内容稿是给创始人看的中文稿，最终用途是帮他打磨思路、并对照着讲 PPT。它不是审阅报告，不是 Fact Bank dump，也不是 PPT 的压缩版。

**金标准样例：** `examples/wanaka-content-draft-golden.md`。生成内容稿时优先参照它的结构、密度和语气。

## 必须的整体结构

按下面顺序，缺一不可：

```
项目亮点（3-4 条）
1. 项目摘要
2. 用户、痛点与现有替代
3. 产品方案与核心流程
4. AI 与关键技术杠杆
5. 为什么现在
6. 市场规模与切入口
7. 竞争格局与替代方案
8. 核心壁垒与可投理由
9. 验证、数据与 GTM
10. 团队与 Founder-Market Fit
11. 商业模式、融资与里程碑
12. 愿景与终局
```

注：18 Part 是底层信息来源，**不要**作为内容稿目录裸列出来。12 个模块是面向创始人的呈现单位。

## 项目亮点

放在所有模块之前。3-4 条。每条由一个**判断性小标题**加一段 100-200 字解释构成。

要求：

- 抓住项目最不可替代、最值得被记住的几个点。
- 标题是判断，不是描述。"团队对北美年轻用户和社交产品有长期积累" 优于 "团队介绍"。
- 解释段直接讲清楚这个判断为什么成立，引用具体事实——经历、技术选择、用户行为洞察。
- 不要写成评审意见、不要给打分。

参照样例：见 Wanaka 金标准开头的"项目亮点"四条。

## 每个模块的写法

### 起手：一句简单、直接、准确的总句

每个模块以一句话开头，定下这一节的判断。例如：

> Wanaka 的产品方案是用 AI 降低游戏创作门槛，同时保留图形化界面，让用户能控制和调整自己的创作。

这句话不要用 punchline / takeaway / 项目亮点等内部词标记。它就是这一节的第一句。

### 展开：事实、依据、解释

总句之后展开 200-500 字。结构灵活，但必须包含：

- 来自材料或问答的**事实**——用户原话、具体场景、数字、对比。
- **依据**——为什么这件事成立，机制是什么，证据在哪里。
- **解释**——这件事对投资人意味着什么、对产品意味着什么。

如果一节信息密度高，可以用 1-3 条 bullet 拆开。如果是流程类（产品方案、Why Now），可以用 5 步以内的编号步骤。

### 收尾：「这里建议补充」

每节末尾**可选**附一段标记为 `**这里建议补充：**` 的灰度说明。它指出这一节如果想更扎实，缺什么数据 / 截图 / 用户故事 / 对比。

要求：

- 用 `**这里建议补充：**` 作为开头标记。
- 1-3 句话，具体、可执行。
- 不要写成对项目本身的批评，而是写成"如果想让这一节更强，需要补 X / Y / Z"。

参照样例：Wanaka 每一节末尾的"这里建议补充"。

## 语气

整体接近**42章经式的投资 memo**——观点清楚、事实支撑、语言干净。不要像咨询模板，不要像机器摘要，不要像英文直译中文。

### 应该做到

- 直接讲清楚项目在做什么、靠什么成立、值得记住的判断是什么。
- 引用材料和问答里的具体内容，让读者能看到事实背后的来源。
- 区分**已经成立的判断**和**还需要补充的部分**。后者通过"这里建议补充"承接，不要混进正文叙事。
- 在合适的地方加一句直接的观察，例如"Wanaka 想做的是把这两件事接起来"。
- 默认中文。英文版只在用户明确需要时生成。

### 不能做

- **不能编造**用户没说过的客户、收入、留存、市场规模、融资金额、团队履历。
- **不能反复复制**用户原文。引用是为了支撑判断，不是凑字数。
- **不能写成审计报告**——"已覆盖"、"部分覆盖"、"缺失"、"投资人可能质疑"这些状态字不要出现。
- **不能出现过程痕迹**——"现有材料里最值得被记住的点"、"采访里有一个很重要的判断"、"根据材料可以看出"、"我们注意到"等等。这些是 agent 的内部语言，不是创始人能直接拿去讲 PPT 的句子。
- **不能用 punchline / takeaway / synthesis** 等内部词作为小节标题或标签。
- **不能把"需要补充"混进投资人版叙事里**。需要补充的部分单独用"这里建议补充"承接。

## 与 Slide Spec 的关系

内容稿和 Slide Spec 共享同一份 Fact Bank，但呈现完全不同：

- 内容稿写给**创始人**，目标是帮他想清楚 + 对照着讲。语言可以叙述、可以有 200-500 字段落。
- Slide Spec 写给**投资人**，目标是 10 秒看懂每页判断。语言必须 slide-ready——主句、必含事实、视觉形式。

内容稿不要把 slide 主句直接搬过来用。内容稿是叙述，slide 是论断。

## 项目不成熟时

如果多个模块的事实仍然很薄（多个 Part 是 missing 或 risky），仍然生成完整 12 模块的内容稿，但：

- 每个薄弱模块的总句要诚实——"Wanaka 目前的市场切入口还在打磨阶段" 优于硬编一个像样的切入口。
- "这里建议补充"段落写得更长，列出需要确认的关键假设。
- 整体内容稿开头加一句话："以下内容是基于当前材料和问答的初版梳理，多个判断仍需 Founder 进一步验证。"

不要硬包装成成熟 BP。

## Checklist

生成内容稿之前最后检查一遍：

- [ ] 开头有"项目亮点"，3-4 条，每条标题是判断。
- [ ] 12 个模块齐全，顺序正确。
- [ ] 每个模块以一句准确的总句开头。
- [ ] 没有任何 `Fact Bank`、`source_material`、`punchline`、`takeaway`、`已覆盖`、`partial` 等内部词。
- [ ] 没有"根据材料可以看出"、"采访里有一个判断"等过程语言。
- [ ] 没有编造的数字、客户、留存、收入、团队履历。
- [ ] "这里建议补充"只出现在每节末尾的独立段落里，不混进主叙事。
- [ ] 语言密度接近 Wanaka 样例，不写水文。
- [ ] 默认中文。
